# P3.4 Record Cleanup & Personal History Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the violation-photo lifecycle (cloud-storage cleanup + audit trail) and surface the user's own check-in history inside the App, so that the data flow becomes:

`App 打卡 → tourism_markers.checkedBy[] → 后台审核 → (违规) 删除 + 物理清理 + 审计 → App 我的打卡`

**Architecture:**

- Cloud storage files live alongside `tourism_markers.checkedBy[].photoCloudURL`. Today P3.3 only removes the array entry, not the underlying file. P3.4 introduces an opt-in `photo-center.deletePhoto(cloudURL)` helper, called by `admin-center.deleteCheckinRecord` and by `marker-center.deleteCheckin` when feature-flagged.
- A new `tourism_audit_logs` collection captures who deleted what, when, and why, both for self-deletion and admin violation-deletion. Logs are append-only.
- App-side: a new "我的打卡" page reads the user's own `checkedBy[]` slices from cloud (admin-center is for admins only, so we extend `marker-center` with a logged-in-user-only `getMyCheckins()` method).

**Tech Stack:** uni-app x / UTS 5.07, uniCloud cloud objects on Aliyun, Vue3 H5 `uni-admin`, HBuilderX Android/Web verification, Node.js syntax + unit tests.

## Task 1: Cloud-storage Photo Deletion

**Files:**
- New: `uniCloud-aliyun/cloudfunctions/photo-center/index.obj.js`
- Modify: `uniCloud-aliyun/cloudfunctions/admin-center/index.obj.js`
- Modify: `uniCloud-aliyun/cloudfunctions/marker-center/index.obj.js`

- [ ] **Step 1: 实现 `photo-center.deletePhoto(data)`**

只接受单条 `cloudURL` 字符串，校验属于本服务空间命名空间，再调用 `uniCloud.deleteFile({ fileList: [cloudURL] })`。返回 `{ deleted, errMsg }`。

- [ ] **Step 2: admin-center 在删记录时可选物理清理**

`deleteCheckinRecord` 接受可选 `purgePhoto: boolean`，默认 `false`。当为 `true` 且 entry 有 `photoCloudURL` 时，先 `tourism_markers` 删条目，再调用 `photo-center.deletePhoto`。物理删除失败不影响数据库删除成功的语义，但要写入审计日志的 `purgeError` 字段。

- [ ] **Step 3: marker-center 用户自删默认不删图**

第一版默认不让普通用户触发物理删除，避免误删后无法找回。可加 hidden flag `purgePhoto: true` 给后续"我的打卡"页面手动触发，但要再过一次 actionSheet 二次确认。

- [ ] **Step 4: 测试**

新增 `photo-center` 单测覆盖 URL 命名空间校验；`admin-center` 集成 stub 测试 purgePhoto 路径。

## Acceptance Criteria

- 后台违规删除时可选"同步删除云端照片文件"，删除完成后该照片 URL 在云存储控制台不再可访问。
- `tourism_audit_logs` 至少包含 admin 违规删除与 user 自删两类记录；后台审计页能按 `type` 过滤。
- App "我的打卡"页能列出当前 uid 在云端的全部 entry，按时间降序，点击可跳回地图聚焦该 marker。
- 用户在我的打卡页删除记录后，地图详情面板与全局人数同步刷新（与 P3.3 一致）。
- 所有新增云对象方法都通过 `node --check` 与单元测试。
- 既有 P3.1 ~ P3.3 acceptance 全部继续保持。
