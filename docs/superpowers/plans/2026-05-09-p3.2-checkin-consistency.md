# P3.2 Check-In Consistency Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make App, uniCloud, and uni-admin consistently agree on per-user check-in state, global check-in counts, task completion, and historical local check-in repair.

**Architecture:** Treat `tourism_markers.checkedBy[]` in uniCloud as the source of truth for global check-in records. The App may keep local state for responsiveness, but must derive “current user checked” from the logged-in uni-id uid and cloud `checkedBy[]`. Historical repair is performed only through authenticated cloud objects, is idempotent by `markerId + uid`, and never lets the client write arbitrary `checkedBy[]`.

**Tech Stack:** uni-app x / UTS 5.07, uniCloud cloud objects on Aliyun, Vue3 H5 `uni-admin`, HBuilderX Android/Web verification, Node.js syntax checks for cloud objects.

## Task 2: Add Idempotent Historical Repair Endpoint

**Files:**
- Modify: `uniCloud-aliyun/cloudfunctions/marker-center/index.obj.js`

- [ ] **Step 1: Add `repairCheckin(data)` to `marker-center`**

Implement only current-user repair. Do not trust client-provided `userId`.

```js
async repairCheckin(data) {
  if (!this.auth.uid) return { errCode: -1, errMsg: '请先登录' }
  const payload = data || {}
  const markerId = Number(payload.markerId)
  if (!markerId) return { errCode: -1, errMsg: '缺少打卡点ID' }

  const markerRes = await col.where({ id: markerId }).limit(1).get()
  if (!markerRes.data.length) return { errCode: -1, errMsg: '打卡点不存在' }
  const marker = markerRes.data[0]

  const checkedBy = marker.checkedBy || []
  const exists = checkedBy.some(entry => entry.userId === this.auth.uid)
  if (exists) {
    return { errCode: 0, errMsg: '记录已存在', data: { repaired: false, existed: true } }
  }

  const checkedEntry = {
    userId: this.auth.uid,
    checkedAt: Number(payload.checkedAt || Date.now()),
    photoCloudURL: payload.photoCloudURL || null,
    note: payload.note || null,
    repaired: true
  }

  await col.doc(marker._id).update({
    checked: true,
    checkinCount: db.command.inc(1),
    checkedBy: db.command.push([checkedEntry]),
    updatedAt: Date.now()
  })

  await incrementUserStats(this.auth.uid, {
    totalCheckins: 1,
    totalPhotos: payload.photoCloudURL ? 1 : 0
  })

  const completedTasks = await checkTasksForMarker(this.auth.uid, marker)
  return { errCode: 0, errMsg: '补传成功', data: { repaired: true, completedTasks } }
}
```

## Task 3: Detect And Repair Missing Current-User Cloud Records

**Files:**
- Modify: `utils/cloudSync.uts`
- Modify: `stores/useMarkerStore.uts`

- [ ] **Step 1: Add a typed diff helper**

```ts
export function findRepairableCheckins(localList: CheckinMarker[], cloudList: CheckinMarker[], uid: string): number[] {
  const ids: number[] = []
  localList.forEach((localMarker: CheckinMarker): void => {
    if (localMarker.checked == false) return
    const cloudMarker = cloudList.find((item: CheckinMarker): boolean => item.id == localMarker.id)
    if (cloudMarker == null) return
    const checkedBy = cloudMarker.checkedBy
    const exists = checkedBy != null && checkedBy.some((entry: CheckinEntry): boolean => entry.userId == uid)
    if (!exists) ids.push(localMarker.id)
  })
  return ids
}
```

- [ ] **Step 2: Add a repair runner**

```ts
export async function repairMissingCheckins(uid: string): Promise<number> {
  const cloudList = await syncMarkers()
  const localList = loadLocalMarkersForRepair()
  const repairIds = findRepairableCheckins(localList, cloudList, uid)
  if (repairIds.length == 0) return 0

  const markerApi = uniCloud.importObject('marker-center')
  for (let i = 0; i < repairIds.length; i++) {
    const marker = localList.find((item: CheckinMarker): boolean => item.id == repairIds[i])
    if (marker == null) continue
    await markerApi.repairCheckin({
      markerId: marker.id,
      checkedAt: marker.checkedAt,
      photoCloudURL: marker.photoCloudURL,
      note: marker.note
    })
  }

  await syncMarkers()
  return repairIds.length
}
```

If `loadLocalMarkersForRepair()` does not exist, implement it in `stores/useMarkerStore.uts` as a read-only snapshot helper that returns markers from local storage using `JSON.parse<CheckinMarker[]>()`.

async getSyncDiagnostics() {
  const markersRes = await db.collection('tourism_markers').get()
  const usersRes = await db.collection('uni-id-users').get()
  let checkinTotal = 0
  let markerWithCheckins = 0

  markersRes.data.forEach(marker => {
    const count = Array.isArray(marker.checkedBy) ? marker.checkedBy.length : 0
    checkinTotal += count
    if (count > 0) markerWithCheckins += 1

  return {
    errCode: 0,
    data: {
      markerTotal: markersRes.data.length,
      markerWithCheckins,
      checkinTotal,

## Next Iteration After P3.2

The recommended next milestone is **P3.3: Check-In Record Management And Photo Review**.

P3.3 should stay focused on check-in record lifecycle and admin review before moving to route/story features.

### P3.3 Goals

- App: when the current user has already checked in at a marker, show the marker as checked and replace the primary action with `删除打卡`.
- Cloud: add an authenticated, idempotent current-user delete endpoint that removes only the current user's `checkedBy[]` entry for `markerId + this.auth.uid` and safely decrements `checkinCount`.
- Admin: redesign the check-in record page so one marker groups all of its check-in records instead of repeating marker information once per record.
- Admin: add larger thumbnails and a preview button/modal so photos can be reviewed before later违规删除 work.
- Admin: simplify the back button to an icon-only control in the top-left corner.

### P3.3 Product Decisions To Confirm

- Deleting a user's check-in removes that user's `checkedBy[]` entry and decrements `checkinCount`; task progress rollback is out of scope for the first version.
- Admin违规删除 should initially remove the offending check-in record from `checkedBy[]`; cloud storage file deletion can be a follow-up unless a safe `photo-center` delete path already exists.
- Photo review should prioritize reviewer ergonomics: large preview, useful metadata, and fast navigation between records.

- App: `pages/index/index.uvue`, `stores/useMarkerStore.uts`, `utils/cloudSync.uts`, `types/marker.uts`

### P3.3 Acceptance Criteria

- After deletion, App and admin agree on `checkinCount` and the current user's checked state.
